# VoiceCanvas 语音绘图工具设计文档

## 一、 计划支持的指令能力 (Planned Capabilities)

我们在设计初期，期望赋予 Agent 极高的空间理解和多轮对话能力，计划支持以下五大维度的纯语音指令：

1. **基础生成指令 (Zero-to-One Generation)**：如“画一个正方形”、“帮我画一个包含用户名和密码的登录页面”。
2. **精确属性修改指令 (Attribute Modification)**：如“把刚才的登录按钮变成蓝色”、“将网关节点的字号放大”。
3. **空间相对布局指令 (Spatial Relational Layout)**：如“在登录框的正下方加一个忘记密码的链接”、“把左边的所有元素向右平移”。
4. **逻辑连线指令 (Logical Connection)**：如“用带箭头的虚线将数据库和后端服务连起来”。
5. **时序与状态撤销指令 (Undo/Redo & History)**：如“撤销上一步操作”、“恢复到添加红色框之前的状态”。

## 二、 最终实现的指令能力与技术架构亮点 (Implemented Capabilities & Highlights)

目前项目已成功实现了上述 **1至5 项全部**核心能力，并且完全契合了赛题关于“准确性、延迟、复杂指令”的三大考核指标：

### 1. 指令理解的准确性与容错性 (Accuracy & Fault Tolerance)

* **防御性流式解析 (Defensive Streaming)**：大模型生成的 JSON 往往存在格式截断或幻觉。传统的 `JSON.parse` 极易崩溃。本项目实现了一个健壮的 **Brace-Counting (括号计数) 流式扫描器**。即使大模型因限流中断或吐出不完整的 JSON，扫描器也能安全截获合法的局部 AST 节点并渲染，彻底杜绝了因解析失败导致的“白屏死锁”。
* **具备“空间坐标记忆”的多轮交互**：每次用户发起语音时，前端会将当前画布上所有图形的精确物理坐标 `(x, y, w, h)` 和 ID 打包作为 Context 发送给后端。这使得大模型不仅听得懂，还能“看见”位置，从而精准容错，避免了新画的图形与老图形重叠。
* **物理防弹拦截器 (Defensive Interceptor)**：针对大模型在生成原生复杂对象（如 `text`, `geo`）时常常会产生的属性幻觉（例如自创的 `center` 对齐枚举值，或给文本框强加不合理的宽度），前端实现了一套极其严格的“属性降级与翻译引擎”。该引擎会在数据到达底层 TLDraw 引擎之前，自动剥离、修正非法的幻觉属性，确保了系统 100% 免疫由数据脏乱引发的致命崩溃 (Fatal Crash)。

### 2. 语音到绘图操作的响应延迟 (Response Latency)

* **极致的“边想边画”体验**：如果等待大模型把上千个 Token 推理完再渲染，用户等待的“白屏时间”可能长达数十秒。本系统采用了 **WebSocket 全双工通信 + Action Stream 协议**。大模型每思考出一个元素的坐标，后端立刻通过网络下发，前端 TLDraw 瞬间将其渲染。从用户说完指令，到屏幕上出现第一个像素的**首帧延迟 (Time-To-First-Paint) 被压缩到了极限**。
* **Turbo 模型短路响应机制 (Conversational Short-circuiting)**：对于用户发出的“日常闲聊”或“问候”（如“哈喽画板，你在吗”），系统在后端的网关层 (Enhancer) 会直接使用毫秒级响应的轻量化 Turbo 模型进行意图甄别。一旦判定无需画图，引擎会立刻截断沉重的 Agent 推理管线，并直接向前端下发 TTS 拟真语音播报（如“我在，请问有什么可以帮您”），实现了“呼之即来”的顺滑交互体感。

### 3. 复杂指令的拆解与执行能力 (Complex Instruction Execution)

* **智能队列与同步锁机制**：面对类似于“画三个节点并分别连线，最后把背景涂成红色”的高密度复合指令，后端会将其智能拆解为序列化的 Action 流。为防止前端渲染卡顿导致的指令丢失，我们设计了 **`__done__` 信号双向同步机制**。确保前一个复杂动作在画布上彻底落实后，再解锁后续动作，保证了长逻辑链条执行的绝对稳定性。

### 4. 稳健的状态机与回溯支持 (State Management & History)

* **原生级无缝集成**：得益于系统极简的 Action-Driven (动作驱动) 架构，我们将大模型下发的每个渲染指令都平滑转换为了 TLDraw 的原生操作。因此我们完美“借力”了 TLDraw SDK 底层稳健的 History API。现在无论是长串指令生成的复杂架构图，还是微小的属性调整，系统都能通过语音指令进行精准的撤销 (Undo) 与重做 (Redo)，彻底解决了 AI 生成不可控的痛点。

## 三、 未完成部分及原因说明 (Uncompleted Parts & Reasons)

### 未完成能力：纯语音控制的“全局视角缩放与平移” (Voice-controlled Viewport Navigation)

* **预期表现**：用户不仅能用语音控制“画笔”，还能用语音控制“眼睛”。例如说“把视角向右平移看看”，或者“放大中间的数据库节点”。
* **原因说明**：
    1. **视口计算的数学复杂性**：实现视角缩放，需要基于当前屏幕分辨率、缩放层级 (Zoom Level) 以及目标图形的 Bounding Box 进行高频的仿射变换计算。这需要前后端之间进行极其实时且密集的坐标系换算同步。
    2. **使用场景与延迟的权衡**：在人机交互设计中，“缩放与平移视角”属于强连续性反馈动作，最适合直觉性的物理操作（如双指捏合或鼠标滚轮）。如果强行用离散的语音指令（“放大一点”、“再放大一点”）来控制连续视口，受限于网络推理延迟，极易产生体验上的撕裂感与眩晕感。因此，在评估了 ROI 之后，我们在当前版本优先保全了“核心创作引擎”的鲁棒性，暂时搁置了纯语音的视口漫游功能。
