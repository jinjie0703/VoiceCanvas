# 提示词工程 (Prompt Engineering)

本项目的核心业务逻辑处理高度依赖大语言模型，因此对 System Prompt 的微调和重构会直接影响系统的稳定性与“聪明程度”。相关提示词存放于 `backend/internal/prompts` 目录下，且会随 Go 代码直接编译为二进制资产。

## 核心设计原则

1. **角色边界与输出封锁**：
   必须在 Prompt 顶部明确告知大模型其核心职责是一个专门负责解析指令并输出 JSON Action 序列的后台计算节点，且必须严禁输出多余的对话、客套话或 markdown 块之外的内容，以保证后端解析器不会因为脏数据而崩溃。

2. **空间感知与坐标计算**：
   由于 LLM 缺乏视觉能力，前端会以列表形式传送已存在形状的位置与宽高等几何数据。提示词中需重点声明**防重叠（Collision Avoidance）**和**智能排版**的逻辑，强制模型在给出新坐标时参考上下文中已有的坐标。

3. **防御性拒识**：
   语音拾音容易将背景闲聊录入。提示词内需要配置防御机制：当判断当前语句与绘图、修改、控制画布无关时，应当输出一个空的 actions 列表并静默丢弃，防止在大屏演示时发生“翻车”。

## 调优方向与最佳实践

- **利用 Few-Shot 示例**：如果您希望系统掌握一种新的图形结构（比如生成复杂流程图），在 Prompt 中追加一组标准的输入输出对话范例（Shot）是成功率最高的做法。
- **与 RAG 结合**：对于 TLDraw SDK 特定属性的理解，尽量避免写死在 Prompt 中，而是通过外部知识库（RAG）动态注入到提示词上下文中，以控制 Token 消耗。
