# Create Standard PR Skill

## 触发条件

1. **严格被动触发**：本技能**禁止**智能体在完成局部的开发、重构或调试子任务后自动、主动地触发运行。
2. 只有当用户发出明确的指令（例如“帮我提交代码”、“保存当前工作”、“开始提 PR”或“合并代码”）时，智能体才能触发此流程。
3. **日常开发中**，智能体修改完代码文件后，应保持修改状态在本地工作区，等待用户进一步的联合修改或开发指令，严禁“改一个文件就提交一次”。

## Instructions (执行步骤)

当你需要为用户创建 PR 时（且仅在收到用户明确指示后），请严格按照以下步骤执行：

1. **状态检查与单一职责检查 (防突击与防冗余)**：
   - 运行 `git status` 和 `git diff` 查看当前所有更改。
   - **单一功能与防突击校验**：评估改动范围。如果发现改动文件极多（例如疑似最后一次性导入全部代码的“突击提交”），必须**强制提醒**用户：比赛严禁临尾突击，须保持全周期持续的 PR 提交。如果当前改动包含多个不相关的改动，需拦截并强烈建议用户拆分为多个独立 PR 分步提交（每个 PR 仅做一件事，粒度尽可能细）。
2. **分支安全校验 (No Direct Main Push)**：
   - 获取当前分支名。如果当前处于 `main` 或 `master` 分支，**必须禁止直接提交和推送**。
   - 引导或协助用户切换到遵循命名规范的新分支（格式：`类型/简短描述`，例如 `feat/voice-controls`）。
3. **代码质量与多模块构建验证 (主分支可运行保障)**：
   - 严格进入各自独立模块的目录进行校验：在 `/frontend` 目录下运行 `npm run build`（或通过相关 skill），在 `/backend` 目录下运行 `go build`，确保没有任何致命编译错误。
   - 保障合并后主分支代码仍保持绝对的**随时可运行状态**（确保评委任意时间查看均能复现演示效果）。如果构建报错，尝试修复或中止提交流程并提醒用户。
4. **依赖检查与版权声明评估**：
   - 检查 `frontend/package.json` 或 `backend/go.mod` 是否有改动。若引入了新的第三方库或框架，记录下来，并在后续步骤中提醒用户同步更新到 `README.md`，同时要在 PR 描述中说明原创功能部分。
   - 询问或自行分析是否复用了旧代码片段。若有，准备在 PR 中注明来源。
5. **生成 Commit 并提交**：
   - 根据 `git diff` 内容生成符合 Conventional Commits 规范的 message。注意提交的时间戳必须在比赛所选批次的开始与截止时间之内。
   - 执行 `git add .` 和 `git commit -m "<你的message>"`。
6. **推送到远程**：
   - 执行 `git push origin <当前分支名>`。
7. **创建 Pull Request (严格的描述与一致性校验)**：
   - **二次自我审阅**：在起草之前，必须确保你将要写的 PR 描述与 `git diff` 的实际改动**严格一致**，杜绝空白、虚假或夸大其词，否则会被评委视为无效作品。
   - **起草 PR 描述**：自动草拟符合规范的 PR 描述，必须包含以下核心段落：
     - **PR 标题**：一句话说明本 PR 新增/修改了什么。
     - **功能描述**：说明该功能的作用与使用方式。
     - **实现思路**：简要说明技术选型或核心实现逻辑。
     - **测试方式**：说明如何验证该功能正常运行（例如本地通过了前后端构建校验等）。
     - *(如适用)* **第三方库依赖声明**：列出本次引入的新依赖，并说明哪些部分是自身原创。提示用户去更新 README。
     - *(如适用)* **代码复用声明**：注明复用代码的来源。
     - *(如适用)* **团队分工**：如果是团队项目，说明各自账号的贡献。
   - **自动创建 PR**：
     - 将描述写入本地临时文件 `.agents/temp_pr_body.md`。
     - 运行 `gh pr create --title "<PR标题>" --body-file ".agents/temp_pr_body.md" --base main --head <当前分支名>`，成功后删除临时文件。
     - *注：若无 `gh` 权限，展示描述给用户并引导去网页端创建。*
8. **合并与本地同步**：
   - 运行 `gh pr merge <PR号> --merge` 自动将 PR 合并入主分支（若有冲突提示手动合并）。
   - 合并完成后，在本地执行 `git checkout main` 并运行 `git pull origin main` 同步云端最新提交，保持工作区干净。
