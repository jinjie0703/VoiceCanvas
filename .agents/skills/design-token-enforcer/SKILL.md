# Design Token Enforcer Skill

为确保产品级应用具有极高统一性的“审美护城河”（Vibe），AI 在写 UI 代码时必须受到严格的标记审查。

## 触发条件

1. 用户下达类似“把界面做得好看点”、“优化排版”、“加上现代化的样式”等泛视觉指令。
2. 在编写任何包含 `className`、`style` 属性的前端组件时。

## 执行步骤

1. **禁止幻想色值**：除了特殊的复杂渐变动画外，绝不允许凭空使用类似 `#38bdf8` 这样的硬编码 HEX 值。
2. **强制替换为 Tailwind Tokens**：
   - 所有的背景色、边框、文字颜色必须映射到 Tailwind v4 规范。
   - 项目主打配色只能是：`slate` (灰度/背景)、`sky` (品牌主色/可点击项)、`emerald` (成功状态)、`rose` (警告/录音状态)。
   - **绝对红线**：一旦代码中出现了 `violet`、`purple` 或 `fuchsia` 等不符合规范的紫色调，立刻自我阻断并重写。
3. **圆角与留白审查**：
   - 在构建卡片时，使用 `rounded-xl` 或 `rounded-2xl`，抛弃生硬的直角。
   - 使用柔和的阴影（如 `shadow-sm`, `shadow-[...]` 自定义低透明度阴影）。
4. 代码输出前，自我校验一遍：这段 UI 的“Vibe”足够专业和克制吗？符合硅谷企业级风格吗？如果确认符合，再提交给人类。
