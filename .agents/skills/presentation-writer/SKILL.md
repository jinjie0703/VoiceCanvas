# Presentation Writer Skill

## 🎯 目标与适用场景

当用户需要准备项目路演（Pitch）、黑客松答辩或技术分享时触发此技能。
本技能不仅负责生成结构清晰的“逐字稿”，更强调通过**“讲故事（Storytelling）”**与**“穿插演示（Interleaved Demo）”**的方式，将复杂的工程实践拆解为有逻辑层次、贴近真实开发场景的叙述。

## 📜 执行规范 (Instructions)

当触发本技能时，请摒弃过分浮夸的词藻（如“护城河”、“秒杀”、“爆肝”等），采用**“务实、从容、娓娓道来”**的专业开发者语调，按以下流程与用户共创：

### 阶段 1：定调与叙事框架 (Discovery & Framing)

1. **倾听诉求**：确认用户的演讲时长、受众背景（偏工程实现还是偏产品体验），以及最想传达的一个核心理念（Takeaway）。
2. **建立层次 (Layering)**：引导用户按照“由表及里”的层次来构建演讲：
   - **表层（起）**：我们在日常开发中遇到的真实痛点（为什么做）。
   - **交互（承）**：用直观的 Demo 展示我们的解法（怎么用）。
   - **底层（转）**：为了实现这个解法，我们在幕后踩了哪些坑，做了哪些工程取舍（怎么做）。
   - **反思（合）**：在这个过程中，我们对 AI 协作开发带来了哪些新思考（收获与延展）。

### 阶段 2：务实的技术亮点提取 (Pragmatic Tech Extraction)

不要干瘪地罗列技术栈，而是将 VoiceCanvas 的技术点转化为“遇到问题 -> 思考 -> 解决”的工程实践：

- **从“等待加载”到“边想边画”**：讲解为何放弃传统的 JSON 全量解析，转而实现基于 Brace-Counting 的流式扫描器。重点讲述“降低首帧延迟”对用户体验的价值。
- **长连接与长推理的并发冲突**：娓娓道来在 WebSocket 全双工通信中遇到的“读写死锁”问题，以及我们如何通过剥离网络 I/O 与引入 `ClientSession` 任务队列来优雅地化解竞态条件。
- **让 Agent 获得“空间感”**：不吹嘘概念，而是务实地讲解前端是如何将几何坐标 (x, y, w, h) 打包回传，从而让 AI 做到“不重叠、懂布局”的多轮交互。
- **不信任原则与防御性编程**：分享我们在面对大模型幻觉时，是如何通过严格的 HTML 实体转义和 RAG (检索增强) 规范来保证系统健壮性的。

### 阶段 3：技术与效果的“穿插式演示”编排 (Interleaved Choreography)

绝对避免“光讲干货没有画面”或者“先干讲 5 分钟再去点屏幕”。必须为用户设计**“边讲边演、所讲即所见”**的穿插式台本：

- **[视觉论证]**：当讲到某项硬核技术时，立刻用一个界面微动作来印证它。例如：“大家看这个生成的瞬间（画面组件弹出），这背后就是我们的流式解析器在起作用……”
- **[视觉动作]**：明确标记鼠标的走向。“此时鼠标不作多余滑动，自然地展开右侧 Debug Logs面板，让大家看清推理流”。
- **[语音输入]**：提供真实、有代表性的自然语言指令示范。
- **[无缝衔接]**：在 AI 渲染的间隙，填补自然的解说词，让现场不会冷场，例如：“在这个短暂的停顿中，Agent 正在计算各个组件的绝对坐标边界……”

### 阶段 4：逐字稿生成与打磨 (Script Generation & Refinement)

- **语气风格**：真诚、务实、像是在和同行交流经验。多用“我们发现……”、“为了解决这个问题，我们尝试了……”等拉近距离的句式。
- **内容把控**：长句拆短，逻辑连接词清晰（首先、然而、因此）。确保每讲完一个技术点，都能回归到“这给用户/系统带来了什么实际收益”。
- **伴生开发提醒**：在打磨稿件时，如果发现讲述某段逻辑时缺少视觉支撑，可以主动建议用户：“我们要不要在这里加一个微小的状态提示（如涟漪动画）或者日志高亮？这样在我们‘穿插演示’时，画面表现力会强很多。”
